## 1. Overview

Build an autonomous trading bot that trades crypto spot pairs on Binance using the
ai-forecasting ensemble models as its signal source. The bot streams prices in real
time, makes swing-frequency trading decisions (positions held hours to days), and is
governed by hard risk limits and a staged rollout: **no live capital is deployed until
the system has proven itself in backtesting and a minimum four-week paper-trading
period.**

This document is the agreement on what we are building, how safety is enforced, and
what "success" and "failure" mean — before any money is involved.

### 1.1 What this is NOT

This is not a guaranteed-profit system. The honest base rate: **most retail algorithmic
trading bots underperform simply buying and holding, after fees and slippage.** The
project is structured so that this question is answered with simulated money, not real
money. If the bot cannot beat buy-and-hold on a risk-adjusted basis in paper trading,
it does not go live. That outcome is a legitimate, useful result — not a failure of the
project.

## 2. Goals and non-goals

### Goals (v1)

1. **G1 — Signal service:** Serve real-time trade signals (direction + confidence) for a
   fixed universe of major crypto pairs, generated by the retrained ensemble models.
2. **G2 — Autonomous execution:** Execute entries, exits, and stop-losses on Binance
   spot without manual intervention, within hard risk limits.
3. **G3 — Full auditability:** Every signal, decision, and order is logged so that
   "why did it buy?" is always answerable, for both stakeholders.
4. **G4 — Real-time monitoring & control:** Live dashboard (positions, P&L, signals,
   model health) and remote control via Telegram, including a kill switch.
5. **G5 — Staged, evidence-gated rollout:** Backtest → paper → small live → scale, with
   explicit go/no-go criteria at each gate (§8).

### Non-goals (v1)

- Futures, margin, or any leverage.
- High-frequency or sub-minute trading.
- Multi-exchange support (Binance only; the abstraction allows adding later).
- Sentiment/news/social data inputs.
- Managing third-party money as a service (see §9.3).
- Mobile app (Telegram covers remote access).

## 4. Product requirements

### 4.1 Intelligence layer (evolves the existing ai-forecasting repo)

- **R1.** Retrain the ensemble (XGBoost, LightGBM, CatBoost) on crypto OHLCV data at
  4-hour and daily bars. TensorFlow and PyTorch are removed from v1 to reduce the
  dependency and image-size problem in the current repo; deep models can return in v2
  if the boosted ensemble proves insufficient.
- **R2.** Fixed trading universe for v1: BTC/USDT, ETH/USDT, SOL/USDT, BNB/USDT,
  plus up to two more majors selected by liquidity. No long-tail coins.
- **R3.** New endpoint `GET /api/v1/signal/{pair}` returning
  `{direction: long|flat, confidence: 0–1, horizon, model_votes, top_features, generated_at}`.
  Spot-only means the bot is long or flat — never short.
- **R4.** Data ingestion from Binance klines (REST backfill + websocket keep-current)
  into PostgreSQL; feature computation on candle close.
- **R5.** Scheduled retraining (weekly, Celery) with walk-forward validation; a new
  model version is promoted only if it outperforms the incumbent on the validation
  window. Model versions are recorded with every signal they produce.
- **R6.** Model-health metrics exposed: rolling directional accuracy, calibration of
  confidence scores, prediction-vs-actual per pair.

### 4.2 Execution layer (Freqtrade)

- **R7.** Freqtrade hosts execution: order lifecycle, partial fills, reconnection,
  position sizing, dry-run mode, backtesting, and Telegram control. We deliberately do
  not hand-roll order management — for live money, battle-tested execution code is a
  safety requirement, not a convenience.
- **R8.** Custom strategy `EnsembleSignalStrategy`: on each 1-hour candle close, fetch
  the signal for each pair; enter long only when (a) signal is long with confidence ≥
  threshold (tuned in backtesting), (b) trend filter passes (price above long EMA), and
  (c) volatility is below a ceiling (no entries into chaos). Exit on opposite signal,
  stop-loss, trailing stop, or max-hold timeout.
- **R9.** If the signal API is unreachable or a signal is stale (> 2 evaluation cycles
  old), the strategy takes **no new entries** and manages existing positions to exit.
  Failure mode is always "stop trading," never "trade blind."
- **R10.** Stop-losses are placed **on the exchange** (stoploss-on-exchange), so open
  positions remain protected even if the bot, VPS, or network dies.

### 4.3 Monitoring and control

- **R11.** Next.js dashboard extended with: open positions and live P&L, equity curve
  vs BTC buy-and-hold benchmark, signal feed with model reasoning, model-health panel,
  and system status (bot heartbeat, API health, data freshness).
- **R12.** Telegram bot (Freqtrade native): status, daily/weekly P&L summaries,
  trade notifications, `/forceexit all` and `/stopbuy` (kill switch) available to both
  stakeholders' Telegram accounts.
- **R13.** Audit log in PostgreSQL: every signal, every decision (entered / skipped and
  which guard blocked it), every order and fill, retained indefinitely.
- **R14.** Alerting: heartbeat monitor (e.g., healthchecks.io) pages if the bot goes
  silent; alerts on circuit-breaker trips, API failures, and abnormal drawdown.

### 4.4 Infrastructure

- **R15.** Runs on a small VPS (not a personal machine), Docker Compose: intelligence
  API, Freqtrade, PostgreSQL, Redis, dashboard. Target monthly infra cost ≤ $20.
- **R16.** Binance API keys are created **without withdrawal permission** — a total
  compromise of the server cannot move funds off the exchange. Keys are IP-restricted
  to the VPS and stored only in server-side env config, never in git.
- **R17.** Automated daily backup of the database (signals, trades, audit log).

## 5. Architecture

```
                   ┌────────────────────────── VPS (Docker Compose) ──────────────────────────┐
                   │                                                                          │
 Binance ──klines──▶  Data ingester ──▶ PostgreSQL ◀── audit log ─────────────┐               │
 (websocket+REST)  │        │                ▲                                │               │
                   │        ▼                │                                │               │
                   │  Intelligence API (FastAPI, ensemble models)             │               │
                   │        ▲     (weekly retrain via Celery/Redis)           │               │
                   │        │ GET /signal/{pair}                              │               │
                   │        │                                                 │               │
 Binance ◀─orders──▶  Freqtrade (EnsembleSignalStrategy)  ──trades/status────▶│               │
 (spot, stop-loss  │        │                                                 │               │
  on exchange)     │        └──REST API──▶  Next.js dashboard  ◀──────────────┘               │
                   │        └──Telegram──▶  both stakeholders (control + kill switch)         │
                   └──────────────────────────────────────────────────────────────────────────┘
```

Data flow, one cycle: candle closes → ingester updates features → Freqtrade asks the
intelligence API for signals → strategy applies guards → orders (if any) go to Binance
with exchange-side stops → decision and orders written to audit log → dashboard and
Telegram reflect state.

## 6. Real-time design

"Real time" in this system means three different clocks, deliberately:

1. **Streaming (continuous):** Binance websockets feed live prices to Freqtrade and
   the ingester. Latency: sub-second, handled by Freqtrade natively.
2. **Decisions (hourly):** Trading decisions happen on 1-hour candle close against
   4h/1d-horizon signals. For swing trading, faster decision loops add fees and noise,
   not returns — this is a feature, not a limitation.
3. **Protection (continuous, exchange-side):** Stop-losses rest on Binance itself.
   Position protection does not depend on our infrastructure being up.

Model retraining is the slowest clock (weekly), with promotion gated on validation
performance (R5).

## 7. Risk management

Hard limits, enforced in code, not by discipline:

| Limit | Value (v1) | Enforced by |
|---|---|---|
| Risk per trade | ≤ 2% of portfolio | Position sizing + stop distance |
| Concurrent positions | ≤ 3 | Freqtrade `max_open_trades` |
| Stop-loss per position | Hard stop, on exchange | R10 |
| Daily loss circuit breaker | −5% day → halt all new entries until manual review | Strategy + protections |
| Max drawdown kill | −15% from equity peak → full halt, stakeholder review required to restart | Protections + alert |
| Signal staleness | > 2 cycles stale → no new entries | R9 |
| Key permissions | Trade-only, no withdrawal, IP-locked | R16 |

Kill switch paths (any one suffices): Telegram `/stopbuy` + `/forceexit all`, dashboard
halt button, or revoking the API key at Binance (works even if all our infra is down).

## 8. Rollout gates (go/no-go)

No gate may be skipped. Each gate's result is shared with both stakeholders before
proceeding.

| Gate | Phase | Duration | Go criteria |
|---|---|---|---|
| **G0** | Walk-forward backtest (≥ 2 years, fees + slippage modeled) | — | Beats BTC buy-and-hold on risk-adjusted return (Sharpe), max drawdown < 25%, positive after fees |
| **G1** | Paper trading (Freqtrade dry-run, live data, live signals) | ≥ 4 weeks | Live paper results within tolerance of backtest expectations; no critical operational failures; uptime ≥ 99% |
| **G2** | Live, small capital (~$500, owner's funds only) | ≥ 4 weeks | Execution matches paper (slippage within modeled bounds); risk limits verified firing correctly in production |
| **G3** | Scale (additional capital, cousin's approval required) | ≥ 3 months at G2 first | Sustained risk-adjusted outperformance; explicit written sign-off from both stakeholders |

**Failure is a valid outcome at every gate.** If G0 or G1 fails, the project stops or
returns to research — with zero money lost. That is the system working as designed.

## 9. Success metrics and honest risks

### 9.1 Success metrics

- **Primary:** Risk-adjusted return (Sharpe) vs BTC buy-and-hold over the same window.
- Max drawdown vs the 15% kill threshold.
- Directional accuracy of signals (target: > 55% on 4h horizon — modest and honest;
  the README's legacy "MAPE < 5% / 60%+" figures are unvalidated and are superseded
  by measured results).
- Operational: uptime ≥ 99%, zero unprotected positions, zero risk-limit violations.

### 9.2 Honest risk disclosure

- **Market risk:** Total loss of deployed capital is possible. Crypto drawdowns of
  50%+ happen regularly; the bot's limits reduce but cannot eliminate this.
- **Model risk:** Backtests overfit. Live performance is usually worse. The G1/G2
  gates exist precisely because of this.
- **Operational risk:** Exchange outages, API changes, bugs. Mitigated by
  exchange-side stops, fail-closed behavior (R9), and monitoring — not eliminated.
- **Counterparty risk:** Funds held on Binance are exposed to exchange failure.
  Only gate-approved capital is kept on-exchange.

### 9.3 Regulatory note

Operating the bot with personal/family funds is generally fine, but **formally managing
someone else's money can constitute regulated investment activity** depending on
jurisdiction and structure. For this project the working arrangement is: each party's
capital stays in accounts they own, or the arrangement stays informal and family-scale.
If this ever grows beyond that, get real legal advice first. This document is not
legal or financial advice.
